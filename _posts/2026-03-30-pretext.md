# Pretext：一个 15KB 的库，凭什么让文字排版快 300 倍？

今天前端圈被一个叫 **Pretext** 的库刷屏了。GitHub 几天就冲到 6800+ stars，Hacker News 首页热议，Simon Willison 专门写了推荐。它的作者是 **Cheng Lou（陈楼）**——React 核心团队前成员、react-motion 的作者、现 Midjourney 前端工程师。

但 Pretext 既不是新框架，也不是 React 的替代品。它只做一件事：**不碰 DOM，纯算术算出文字该怎么排**。听起来平平无奇，但它解决的是一个困扰浏览器整整 30 年的性能瓶颈。

## 一、先搞懂一个问题：浏览器排版文字为什么这么慢？

如果你不做前端，可能从来没想过这个问题。我们换个日常场景来理解。

### 1.1 一个比喻：在白板上排通知

想象你是一个行政，要在一块固定宽度的白板上贴一段通知。你需要知道：

- 每个字多宽？
- 一行能放几个字？
- 到哪里该换行？
- 整段通知有多高？（这样才知道白板够不够大）

你的做法是：**把字一个个贴上去，贴到一行放不下了就换行，最后量一下总高度。**

浏览器做的事情和这完全一样——它必须先把文字实际"放"到页面上，才能知道文字有多高、在哪里换行。这个过程叫 **Reflow（回流/重排）**。

### 1.2 Reflow 为什么是性能杀手？

问题在于：每当你想知道一段文字的高度，浏览器就得走一遍完整的排版流程。就像你每次想知道白板上通知有多高，就得把所有字重新贴一遍。

在真实的网页里，这意味着：

- 你有 500 个文本框需要动态计算高度？浏览器要 reflow 500 次
- 每次 reflow 可能阻塞 10-100 毫秒（在手机上更慢）
- 如果你要做动画——比如文字跟着鼠标动态重排——每秒需要 60 帧，每帧 16 毫秒，根本来不及

用 Google Chrome 团队的话说：**一次强制 reflow 就能让主线程卡住几十毫秒。** 你在网页上看到的卡顿、闪烁、跳动，很多时候就是 reflow 造成的。

## 二、Pretext 的解法：不量了，算！

Pretext 的核心思路惊人地简单：

> **别让浏览器去"排版然后量"——我自己算出来。**

它的做法分两步：

### 第一步：`prepare()` —— 量一次字宽，缓存起来

```javascript
import { prepare, layout } from '@chenglou/pretext'

const prepared = prepare(canvas, "你好世界Hello World", {
  fontFamily: "sans-serif",
  fontSize: 16,
})
```

这一步做了什么？

1. 把文字切成一个个"段"（segment）——用的是浏览器原生的 `Intl.Segmenter`，所以中文、日文、阿拉伯文都支持
2. 用 Canvas API 量每个段有多宽——**Canvas 的 `measureText()` 不触发 reflow**，它直接问浏览器字体引擎"这个字多宽"，不需要真的把字放到页面上
3. 把所有宽度数据缓存起来

这一步只需要做一次，大约 17 毫秒处理 500 个文本块。

### 第二步：`layout()` —— 纯数学计算排版

```javascript
const result = layout(prepared, { width: 300 })
// result.height → 文本高度
// result.lines → 行数
```

这一步做了什么？

1. 拿着缓存的字宽数据
2. 用纯算术计算：第一行能放几个字？在哪换行？一共多少行？总高度多少？
3. **没有 DOM 操作，没有 Canvas 调用，没有字符串拼接——纯粹的加减乘除**

这一步有多快？**500 个文本块只需要 0.05-0.10 毫秒。** 对比 DOM 方案的 15-30 毫秒，快了 **300-600 倍**。

### 为什么能这么快？

用回白板的比喻：

- **传统方式**：每次想知道通知多高，就把所有字重新贴一遍，然后量——慢
- **Pretext 方式**：先量好每个字的大小记在本子上，之后只要知道白板多宽，拿计算器算一下就知道有几行、多高——快

关键的洞察是：**字体确定后，每个字的宽度就是固定的。** 不需要每次都重新量。把量的过程和排的过程分开，排版就变成了纯数学问题。

## 三、它能做什么？几个让人惊掉下巴的 Demo

Pretext 的 demo 页面直接让人明白了"快"意味着什么。

### 3.1 中国龙破文而过

![Dragon Text Reflow Demo](/assets/images/pretext-dragon.gif)

最震撼的 demo：一条 80 段的中国龙跟着你的鼠标在整页文字中穿行。龙经过的地方，文字实时向两边散开；龙走了，文字合拢回来。**60 帧，零卡顿，零 DOM 读取。**

这在以前是不可能的——你需要每帧重新计算所有文字的位置，传统 DOM 方案在第一帧就卡死了。

### 3.2 杂志级排版引擎

![Editorial Engine Demo](/assets/images/pretext-editorial.gif)

文字自动绕过图片、引用块和装饰元素流动，像 InDesign 一样的多栏排版。以前这种效果只能用 PDF 渲染器实现，现在浏览器里就能跑，而且可以交互。

### 3.3 动态折叠/展开

![Accordion Demo](/assets/images/pretext-accordion.gif)

列表项展开收起时，高度变化有流畅动画。这个看似简单，但传统方案要先把内容渲染出来量高度，再设置动画——两步之间必然闪一下。Pretext 直接算出目标高度，动画丝滑。

## 四、一些关键细节

### 4.1 它不是框架

Pretext **不是** React、Vue、Svelte 那样的前端框架。它是一个**纯计算库**——给它文字和容器宽度，它返回排版数据（高度、行数、换行位置）。至于你拿这些数据去渲染 DOM、Canvas 还是 SVG，随你。

```
Pretext = 文本测量 + 排版计算
它不管渲染，只管算
```

### 4.2 多语言支持

这不是一个只能处理英文的玩具。Pretext 内置了：

- **CJK**（中日韩）：正确的字符分割和禁则处理（比如句号不能出现在行首）
- **阿拉伯文**：从右到左排版，标点聚合
- **Emoji**：正确计算宽度（emoji 的宽度和普通字符不一样）
- **混合排版**：中英文混排、多语言混排都能处理

### 4.3 体积和依赖

- **15KB**，gzip 后更小
- **零依赖**
- 纯 JavaScript/TypeScript
- 因为不依赖 DOM，也能在 Node.js、Deno、Edge Functions 里跑

### 4.4 CSS 兼容性

Pretext 支持常用的 CSS 文本属性：

- `white-space: pre-wrap`（保留换行和空格）
- `word-break: normal`
- `overflow-wrap: break-word`
- `line-break: auto`

你可以把它当成一个"CSS 文本排版引擎的纯计算版本"。

## 五、为什么它和 AI 特别配？

Cheng Lou 发布时说了一句话：**"throw your AI into making cool demos for it!"**（把你的 AI 扔进去做 demo 吧！）

这不是营销话术，而是 Pretext 的一个设计目标：**API 简单到 AI 能直接用。**

为什么这很重要？

1. **AI 设计工具需要即时文字尺寸**：当 AI 生成一个界面布局时，它需要马上知道"这段文字在这个宽度下有多高"。传统 DOM 方案太慢了，没法在 AI 的迭代循环里用
2. **API 只有两个核心函数**：`prepare()` 和 `layout()`，参数简单，AI 不容易调错
3. **纯函数，无副作用**：输入确定，输出就确定，AI 很容易理解和调试

## 六、它的局限性

客观地说，Pretext 目前还不完美：

- **SSR 还没实现**：服务端渲染还在计划中。如果你需要在服务器上计算文字排版，暂时还不行
- **初始测量有成本**：`prepare()` 那一步还是需要 Canvas 环境，不是完全零成本。500 个文本块大约 17 毫秒——对于初始加载来说可以接受，但不是零
- **还很新**：发布才几天，生产环境使用需要谨慎评估
- **不能完全替代 CSS**：它算的是纯文本排版，不处理 HTML 嵌套元素、inline 元素混排等复杂场景

## 七、怎么理解它的意义？

Pretext 做的事情，本质上是把浏览器的文字排版能力**解耦**出来了。

30 年来，"知道文字有多高"和"把文字放到页面上"是绑死在一起的。你想知道高度？必须先渲染。想实时计算？必须实时渲染。这个绑定关系限制了所有需要文字尺寸信息的交互设计。

Pretext 把这两件事拆开了：**测量和渲染解耦。** 你可以在不渲染任何东西的情况下，瞬间知道文字的排版结果。这打开了一扇门——凡是以前因为"测量文字太慢"而做不了的效果，现在都有可能了。

它不会替代 React，不会替代 CSS，甚至不会出现在大多数项目里。但对于那些需要高性能文字排版的场景——编辑器、设计工具、数据可视化、AI 界面——它可能是一个改变游戏规则的基础设施。
